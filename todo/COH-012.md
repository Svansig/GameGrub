## Problem

Utility packages risk becoming catch-all layers with high coupling.

## Scope

- In scope:
  - Define allowed dependency directions for utils subpackages.
  - Flag and plan high-risk exceptions.
- Out of scope:
  - Full utils package reorganization in one PR.

## Validation

- [x] Added dependency guardrails for utils packages to ARCHITECTURE.md section 5:
  - Utils should depend on service/domain layers, not vice versa
  - UI-layer utils (`ui/utils/`) should only depend on UI models
  - Business-logic utilities can depend on service/domain but not UI
  - Platform-specific utils should encapsulate platform adapters
