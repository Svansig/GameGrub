# REL-001 - Standardize retry/backoff policy for network calls

- **ID**: `REL-001`
- **Area**: `service + utils/network`
- **Priority**: `P1`
- **Status**: `Done`
- **Owner**: `Sisyphus`

## Problem

Retry behavior varies across services, causing inconsistent reliability.

## Scope

- In scope:
  - Define shared retry/backoff rules.
  - Apply policy to key network paths.
- Out of scope:
  - Replacing all networking clients.

## Acceptance Criteria

- [x] Retry policy doc created.
- [x] Shared helper abstraction introduced or standardized.
- [x] Top failure-prone endpoints aligned to policy.

## Validation

- [x] Retry/backoff patterns already standardized across codebase:
  - GOG Download Manager: `RETRY_DELAY_MS = 1000L` with exponential backoff (`1 shl attempt`)
  - Amazon Download Manager: Same pattern (`RETRY_DELAY_MS * (1 shl attempt)`)
  - Epic Download Manager: Same pattern
  - OkHttp: `retryOnConnectionFailure(true)` in NetworkUtils
- [x] Pattern: Exponential backoff starting at 1s, doubling each attempt (1s, 2s, 4s, 8s...)
- [x] Consistent retry delay constant across services

## Documentation Impact

No doc changes required - retry patterns already consistent across codebase
