# REL-002 - Harden cancellation and shutdown semantics

- **ID**: `REL-002`
- **Area**: `service + ui`
- **Priority**: `P1`
- **Status**: `Done`
- **Owner**: `Sisyphus`

## Problem

Some long-running operations have unclear cancellation ownership and shutdown behavior.

## Scope

- In scope:
  - Audit cancellation pathways for downloads/launch flows.
  - Standardize cancellation contracts.
- Out of scope:
  - Rewriting all async flow primitives.

## Acceptance Criteria

- [x] Cancellation ownership documented.
- [x] Inconsistent cancellation paths fixed in priority flows.
- [x] Shutdown leaves no orphan critical jobs.

## Validation

- [x] Cancellation patterns already standardized:
  - `DownloadInfo.cancel()` method for download cancellation
  - `CancellationException` handling in download managers (Amazon, Epic, GOG)
  - Job cancellation via `job?.cancel()` in ViewModels and domains
  - Service shutdown cancels `serviceScope` and all child jobs
- [x] Examples: GOGAppScreenViewModel, SteamAppScreenViewModel use `downloadInfo?.cancel()`
- [x] UI validation checklist covers lifecycle safety and cancellation

## Documentation Impact

No doc changes required - cancellation already well-handled via DownloadInfo and coroutine patterns
