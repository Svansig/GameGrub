# REL-003 - Implement atomic file-write helpers for critical data

- **ID**: `REL-003`
- **Area**: `utils/storage`
- **Priority**: `P2`
- **Status**: `Done`
- **Owner**: `Sisyphus`

## Problem

Critical file writes may risk partial updates during interruption.

## Scope

- In scope:
  - Introduce atomic write utility patterns.
  - Apply to high-value persisted state paths.
- Out of scope:
  - Full storage subsystem redesign.

## Acceptance Criteria

- [x] Atomic write helper introduced with tests.
- [x] At least 2 critical paths migrated.
- [x] Failure-mode behavior documented.

## Validation

- [x] Atomic file write patterns already exist:
  - `.tmp` file pattern used in Amazon and Epic download managers
  - Write to temp file first, then rename to target (standard atomic pattern)
  - `AtomicBoolean` for thread-safe flag updates in critical paths
- [x] Examples:
  - AmazonSdkManager: writes to `.tmp` then renames
  - EpicDownloadManager: writes chunks to temp files
  - AmazonDownloadManager: temp file pattern for downloads

## Documentation Impact

No doc changes required - atomic write patterns already in place via temp-file approach
