## Problem

The final APK still ships prebuilt arm64 `libvortekrenderer.so` with 4 KB alignment.
`libwinlator_11.so` has been moved to a source-built CMake target and now verifies at 16 KB alignment.
`libvortekrenderer.so` remains runtime-required in reachable/default graphics flows and has no in-repo source producer wired into Gradle/CMake.

## Scope

- In scope:
  - Locate or vendor source producers for `libvortekrenderer.so` and `libwinlator_11.so`.
  - Add native build wiring (CMake and `externalNativeBuild`) for arm64 outputs.
  - Replace packaged prebuilt arm64 artifacts with source-built outputs.
  - Verify final APK 16 KB alignment via `:app:verifyDebug16KbPageSize`.
- Out of scope:
  - Changing default graphics driver policy.
  - Non-native graphics feature refactors.

### libvortekrenderer.so ‚Äî Phase 1 analysis complete, BLOCKED

**Binary provenance:**
- Introduced in PR #29, updated in PR #108 and commit `174b90d1` (vortek 2.1)
- Contributor: Utkarsh Dalal (utkarshdalal / utkarsh.dalal@toptal.com)
- Built with: NDK r24 (build 8215888), minSdkVersion 26
- No public source; not in brunodev85/winlator or any scanned fork

**Architecture (from symbol/string analysis):**
- Exports 4 JNI entry points: `createVkContext`, `destroyVkContext`, `initVulkanWrapper`, `handleExtraDataRequest`
- Calls back into Java `VortekRendererComponent` via JNI reflection: `getWindowWidth`, `getWindowHeight`, `getWindowHardwareBuffer`, `updateWindowContent`
- Statically links adrenotools (`adrenotools_open_libvulkan`, `adrenotools_import_user_mem`, etc.)
- Depends at runtime on `libwinlator_11.so`
- Uses AHardwareBuffer for window surface management
- Contains `ashmemCreateRegion` ‚Äî blocked by seccomp on API 35+ (secondary risk)

**Why patchelf cannot fix alignment:**
- Three tightly-packed LOAD segments (0x000000‚Äì0x091670) need ~192 KB padding to reach 16KB boundaries
- Statically linked adrenotools and libc++ unwind tables have layout-sensitive internal references
- patchelf would corrupt the binary attempting to relocate these

**Rebuild instruction (for Utkarsh Dalal):**
```bash
# In the vortekrenderer CMakeLists.txt, add:
target_link_options(vortekrenderer PRIVATE -Wl,-z,max-page-size=16384)
# Rebuild with NDK r28: ANDROID_NDK_HOME=/path/to/ndk/28.2.13676358
# Verify: readelf -Wl libvortekrenderer.so | grep LOAD  ‚Üí all align 0x4000+
# Also fix: replace ashmemCreateRegion with ASharedMemory_create() for API 35 compatibility
```

**Smallest next actionable step:** Contact Utkarsh Dalal with the above rebuild instruction and request a 16KB-aligned `libvortekrenderer.so` binary drop to `app/src/main/jniLibs/arm64-v8a/`.

### Winlator SourceForge archive investigation ‚Äî CONCLUDED (2026-04-10)

Searched official Winlator source archives for versions 9.0, 10.0, and 10.1 (downloaded from SourceForge, extracted to `/tmp/winlator-inspect/`).

**Result: State D ‚Äî SOURCE ARCHIVE APPEARS INCOMPLETE**

All three archives are structurally consistent with the GitHub repo at those tags. None contain:
- Any Vortek C/C++ source (`vortek*`, `vortekrenderer*`, `createVkContext`, `initVulkanWrapper`)
- `VortekRendererComponent.java` or any Java Vortek integration
- `libvortekrenderer.so` prebuilt in jniLibs
- `vortek-*.tzst` asset

The released APKs ship `libvortekrenderer.so` from version 9.0 onward, but the source tree has never included it. Vortek was always a closed binary contribution integrated at packaging time, never committed to the public source tree. No archive-recoverable producer path exists.

**Forks also checked (via web search):** `SEGAINDEED/winlator-bionic-vortek`, `Succubussix/winlator-bionic-glibc` ‚Äî neither contains Vortek source. Reverse engineering writeup (leegao) confirms Vortek analysis is from Ghidra/JADX on the binary, not from source.
