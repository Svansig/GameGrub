## Problem

Startup logic is difficult to reason about as one large flow.

## Scope

- In scope:
  - Break startup into explicit stage methods.
  - Define stage responsibilities and ordering.
- Out of scope:
  - Startup behavior changes beyond structural refactor.

## Decomposition Plan (Current Pass)

Primary hotspot: `app/src/main/java/app/gamegrub/service/steam/SteamService.kt`

Target startup stages:
1. Service/lifecycle preflight and runtime-state initialization.
2. Client connection + callback wiring.
3. Authentication/session recovery handoff.
4. Library/install domain startup handoff.
5. Post-start observability/health checks.

### Implementation Checklist

- [ ] Extract each stage into named private methods first (no behavior change).
- [ ] Move business-heavy portions of each stage to appropriate domain APIs.
- [ ] Remove direct manager/singleton access from startup path and route through domains.
- [ ] Ensure startup error handling and retry policy remain equivalent.
- [ ] Add focused tests for stage ordering and failure recovery boundaries.
