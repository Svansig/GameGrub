# SRV-027 - Migrate UI download callers to direct domain injection

- **ID**: `SRV-027`
- **Area**: `service/steam + ui`
- **Priority**: `P2`
- **Status**: `Backlog`
- **Owner**: `TBD`

## Problem

After SRV-016a/b/c, download orchestration lives in SteamInstallDomain, but UI callers still go through SteamService.Companion facade. Architecture goal is for new call sites to use domain APIs directly.

## Context

- Current: `SteamAppScreen` calls `SteamService.downloadApp()` which delegates to domain
- Best practice: UI should inject and call `SteamInstallDomain.downloadApp()` directly
- Pattern exists: Other screens use EntryPointAccessors for Hilt injection in Compose

## Scope

- In scope:
  - Inject SteamInstallDomain into SteamAppScreen (via EntryPointAccessors pattern)
  - Update download call sites to use domain directly
  - Remove companion delegation (after verifying direct access works)
- Out of scope:
  - Changes to download logic itself

## Acceptance Criteria

- [ ] SteamInstallDomain injected into SteamAppScreen
- [ ] All download call sites use domain directly
- [ ] Build passes

## Validation

- [ ] Build passes
- [ ] Manual test: download works via direct domain call
