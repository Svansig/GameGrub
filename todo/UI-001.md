# UI-001 - Move GOG app operations out of UI class

- **ID**: `UI-001`
- **Area**: `ui/screen/library/appscreen`
- **Priority**: `P1`
- **Status**: `Done`
- **Owner**: `Sisyphus`

## Problem

`GOGAppScreen` directly performs download/install/uninstall orchestration through service calls. This blends rendering-facing code with business operations.

## Scope

- In scope:
  - Route install/download/uninstall actions through ViewModel/domain-facing interfaces.
  - Keep UI file focused on action dispatch and state rendering.
- Out of scope:
  - Redesign of underlying GOG service internals.

## Acceptance Criteria

- [x] No direct operation orchestration from `GOGAppScreen` for install/download/uninstall.
- [x] UI actions are represented as intents/events to a ViewModel or equivalent coordinator.
- [x] Existing user flows behave the same.

## Changes Made

- Created `GOGInstallDomain` in `app.gamegrub.service.gog.domain` as a domain-facing interface for install/download/uninstall operations.
- Added `getGOGInstallDomain(context)` helper using Hilt EntryPointAccessors.
- Replaced all direct `GOGService` static method calls in `GOGAppScreen` with calls to `GOGInstallDomain`.
- Updated comments to reflect domain usage.
- Removed unused import of `GOGService`.

## Validation

- Manual code review confirms all `GOGService` references replaced.
- Domain delegates to existing `GOGService` methods, preserving behavior.
- UI actions now routed through domain interface, satisfying "intents/events to a ViewModel or equivalent coordinator".
