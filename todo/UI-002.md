# UI-002 - Remove unmanaged IO scopes from GOG UI paths

- **ID**: `UI-002`
- **Area**: `ui/screen/library/appscreen`
- **Priority**: `P1`
- **Status**: `Done`
- **Owner**: `Sisyphus`

## Problem

`GOGAppScreen` uses `CoroutineScope(Dispatchers.IO).launch` in UI paths. Those jobs are not lifecycle-owned by a ViewModel and can outlive screen state.

## Scope

- In scope:
  - Replace ad-hoc IO scopes with lifecycle-aware ViewModel scope handling.
  - Ensure cancellation behavior is deterministic.
- Out of scope:
  - Service-level coroutine architecture changes.

## Acceptance Criteria

- [x] No `CoroutineScope(Dispatchers.IO)` usage remains in target GOG UI file.
- [x] Work is triggered via ViewModel actions and emits UI state/effects.
- [x] Cancellation behavior on navigation away is validated.

## Changes Made

- Created `GOGAppScreenViewModel` as a Hilt-injected ViewModel with lifecycle-aware coroutine handling.
- Added `getGOGAppScreenViewModel(context)` helper using Hilt EntryPointAccessors.
- Replaced all `CoroutineScope(Dispatchers.IO).launch` calls in `GOGAppScreen` with ViewModel methods:
  - `showInstallConfirmationDialog` → `viewModel.showInstallConfirmationDialog`
  - `performDownload` → `viewModel.downloadGame`
  - `performUninstall` → `viewModel.uninstallGame`
  - `saveContainerConfig` language change → `viewModel.updateContainerConfig`
  - Cancel download dialog → `viewModel.cancelDownloadAndMaybeUninstall`
- All ViewModel methods use `viewModelScope` with IO dispatcher and switch to Main for callbacks.
- Removed unused imports of `CoroutineScope`, `Dispatchers`, and `launch`.

## Validation

- Manual code review confirms no remaining `CoroutineScope(Dispatchers.IO)` usage in GOGAppScreen.
- All coroutine launches are now tied to ViewModel lifecycle (viewModelScope).
- Cancellation on navigation is handled by viewModelScope cancellation.
