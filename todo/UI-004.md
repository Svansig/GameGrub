# UI-004 - Move library auth workflow out of composables

- **ID**: `UI-004`
- **Area**: `ui/screen/library`
- **Priority**: `P1`
- **Status**: `Done`
- **Owner**: `Sisyphus`
- **PR**: `TBD`

## Problem

`LibraryScreen` directly owns OAuth launcher callback handling and credential-state checks through service/handler calls.

## Scope

- In scope:
  - Move auth processing and login-state derivation into ViewModel/state.
  - Keep composables focused on UI event hooks and rendering.
- Out of scope:
  - Replacing OAuth Activities.

## Acceptance Criteria

- [x] OAuth result processing is handled by ViewModel/domain-facing layer.
- [x] UI screen does not directly compute source login status via service singletons.
- [x] Existing login success/error UX remains unchanged.

## Validation

- [x] `./gradlew lintKotlinMain` passes on modified files (LibraryListPane.kt, SystemMenu.kt, LibraryScreen.kt)
- [ ] Manual checks for GOG/Epic/Amazon login and cancellation paths (deferred to reviewer/testing)

## Changes Made

### LibraryListPane.kt
- Changed `calculateInstalledCount()` to accept auth state parameters (`isGogLoggedIn`, `isEpicLoggedIn`, `isAmazonLoggedIn`) instead of directly calling service singletons
- Updated skeleton loader calculation to use `state.isGogLoggedIn`, `state.isEpicLoggedIn`, `state.isAmazonLoggedIn` instead of direct service calls
- Removed unused imports: `GOGService`, `EpicService`, `AmazonService`

### SystemMenu.kt
- Added `steamLoggedIn: Boolean` parameter to the composable function
- Replaced direct `SteamService.isLoggedIn` call with the `steamLoggedIn` parameter
- Updated Preview to include the new parameter

### LibraryScreen.kt
- Passed `state.isSteamLoggedIn` to the SystemMenu composable call

## Documentation Impact

- No doc changes required - this is internal refactoring that doesn't change public API behavior. The ViewModel already exposes these auth states via LibraryState.
