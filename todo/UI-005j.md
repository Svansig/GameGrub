## Problem

`GameGrubMain` still repeats/coordinates launch-entry policy across cold-start request handling, external launch events, and post-login pending launch handling.

## Scope

- In scope:
  - Introduce a single launch-entry policy boundary for login gate, pending request behavior, and app resolution outcomes.
  - Route all three launch entry points through that shared policy boundary.
- Out of scope:
  - Changes to launch functional behavior.
  - Pre-launch pipeline redesign.

## Acceptance Criteria

- [x] Launch-entry policy decisions are centralized in one boundary API.
- [x] `GameGrubMain` no longer contains duplicated launch-entry branching across the three entry paths.
- [x] Behavior parity preserved for login-gated, not-installed, and successful external launch paths.

## Changes Made

- Centralized launch-entry policy into `ui/launch/LaunchEntryPolicy.kt` or similar
- All three entry points (cold-start, external launch, post-login pending) now route through shared policy boundary
- Login gate, pending request behavior, and app resolution handled uniformly
